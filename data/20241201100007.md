TCP/IP, HTTP, DNS, sockets. The fundamentals that everything else is built on.

The mental model that made it click: a request goes out, something happens at the other end, a response comes back. That's it. Everything else is just detail about how each step works and what can go wrong.

TCP is reliable: it handles packet ordering, retransmission, congestion control. UDP is unreliable but fast: fire and forget, used for things like video streaming where a dropped frame is better than waiting to retransmit. HTTP runs on top of TCP. HTTP/2 and HTTP/3 add multiplexing and other improvements over the original.

DNS is the phone book. You type `api.example.com`, DNS tells you the IP address. There's caching at every level which is why DNS changes take time to propagate and why "clear DNS cache" fixes weird issues.

The 3-way handshake (SYN, SYN-ACK, ACK) is how TCP connections are established. TLS adds a few more round trips on top of that for encryption negotiation. This is why connection latency matters -- you're paying for those round trips before any actual data moves.

Networking felt abstract until I was debugging a production issue at 2am -- slow requests, intermittent timeouts, couldn't tell if it was the server or the client or something in between. Tools like `curl`, `tcpdump`, checking response headers, understanding what a 502 vs 503 vs 504 actually means -- that's when all of this stopped being academic.
