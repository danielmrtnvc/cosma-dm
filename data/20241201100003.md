Big O notation is the language for talking about how an algorithm scales. Not how fast it runs on your machine right now, but how its runtime grows as the input grows.

The common ones: O(1) constant, O(log n) logarithmic, O(n) linear, O(n log n) most sorting algorithms, O(n²) nested loops, O(2^n) exponential disaster.

The intuition that helped me most: if you double the input size, what happens to the runtime? If nothing changes, it's O(1). If it roughly doubles, it's O(n). If it roughly quadruples, it's O(n²). That's the mental check I run when I'm not sure.

Log n is the one people find least intuitive. The key is that log n grows very slowly -- log₂(1,000,000) is only about 20. So any time you're repeatedly halving the problem (binary search, balanced tree operations), you're in log n territory. That's why binary search feels like magic -- even on a billion elements it only takes about 30 steps.

You don't need to do the full formal analysis every time. Most of the time you can just look at the loop structure. One loop over n: O(n). Two nested loops over n: O(n²). Recursion that halves the input each time: O(log n) or O(n log n) depending on the work per step.

Space complexity matters just as much but gets less attention. Stack depth in recursion is a real concern. Allocating a new data structure inside a loop is the kind of thing that sneaks up on you.
