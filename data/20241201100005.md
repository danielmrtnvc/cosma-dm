CPU, memory hierarchy, cache, registers. This is the layer where software performance is actually determined.

The memory hierarchy is the most important thing to understand. Registers are nanoseconds. L1 cache is a few nanoseconds. L2 cache, more. Main memory (RAM) is maybe 100x slower than L1. Disk is another several orders of magnitude slower. This isn't just trivia -- it changes how you write code if you're thinking about it.

Cache lines are the unit of data movement between cache and RAM. When you access one element of an array, the CPU pulls in a whole cache line (typically 64 bytes). So iterating through an array sequentially is fast because the next element is already in cache. Jumping around in memory (pointer chasing in linked lists, hash map lookups) is slower because you keep missing the cache.

Pipelining means the CPU is executing multiple instructions simultaneously at different stages. Branch misprediction is the thing that breaks pipelining -- the CPU guessed wrong about which branch of an if statement to execute and has to flush its pipeline. This is why some tight loops benefit from branchless code even when it looks more complex.

The moment all of this clicked for me: I started understanding why some "simple" code is actually slow, and some "complex" code is fast. It's not about the algorithmic complexity -- it's about what the hardware is actually doing. Understanding the stack from logic gates up to software changes how you think about performance.
