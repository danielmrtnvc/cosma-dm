Processes, threads, scheduling, memory management, file systems. The OS course was one of the hardest in the program and also one of the most useful in retrospect.

A process is an instance of a running program with its own memory space. A thread is a unit of execution within a process -- threads share memory, which is what makes them fast and also what makes concurrency bugs so painful to debug. The OS scheduler decides which process/thread gets CPU time and for how long.

Virtual memory is the abstraction that lets each process think it has its own address space. The OS, with hardware help (the MMU), handles the mapping from virtual addresses to physical ones. This is also where page faults happen -- when you access a virtual address that isn't currently backed by physical memory.

Deadlock and race conditions: deadlock is when two processes are each waiting for something the other holds. A race condition is when the outcome depends on the order of execution in a way that isn't controlled. Both are the kind of bug that's correct sometimes and broken sometimes, which makes them the most frustrating bugs to track down.

The thing I appreciate in hindsight: almost everything in software engineering comes back to OS concepts. Distributed systems are basically multi-process coordination at network scale. Databases have their own schedulers and memory management. Docker containers are OS-level isolation primitives. Once you understand the foundations, the higher-level abstractions make more sense.
