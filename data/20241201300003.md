The gap between "it works in the playground" and "it works reliably at scale" is where most LLM projects run into trouble.

The things you have to think about when shipping LLM-powered features to real users:

Latency: LLM calls are slow, often 2-10 seconds depending on model and prompt length. That's fine for some use cases and terrible for others. Streaming responses (showing output as it generates) helps the perception problem even if it doesn't change the actual time.

Cost: API calls add up. A feature that costs $0.01 per invocation feels free in testing and becomes significant at scale. You need to think about cost per user action and whether it's sustainable at your target scale.

Hallucinations: models confidently state wrong things. For anything where accuracy matters, you need either retrieval augmentation (give the model the relevant facts rather than relying on its training), validation, or human review in the loop.

Prompt injection: when user input becomes part of a prompt, users can try to override your instructions. "Ignore previous instructions and..." is the classic example. This is a real security concern for anything where the LLM has access to actions or data.

What the Nodalli stack taught me: reliability comes from tight scoping. A model given a narrow, well-defined task with good examples fails much less often than one given a broad, open-ended task. The more constrained the problem, the more predictable the output.
