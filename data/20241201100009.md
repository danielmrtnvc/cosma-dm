VHDL, FPGAs, hardware description languages. A completely different way of thinking about computation.

An FPGA (field-programmable gate array) is a chip you can configure to implement arbitrary digital logic. Instead of running software on a fixed processor, you're defining the actual hardware structure. The "program" you write isn't a sequence of instructions -- it's a description of circuits.

This is the hardest mental shift. Software runs sequentially (or with managed concurrency). Hardware runs in parallel by default. Every logic gate in your design is operating simultaneously. You're not thinking about what happens next, you're thinking about what everything is doing right now.

VHDL (and Verilog) are the languages for this. They look like programming languages but behave differently. A statement like `a <= b AND c` doesn't execute -- it describes a wire. The whole paradigm is about describing connections, not sequences.

The OCT project is where this became concrete. The signal acquisition and initial processing needed to happen at hardware speed, faster than any software could manage. That forced learning to think in hardware, which is a genuinely different cognitive mode than writing code.

I think everyone who's serious about computer engineering should do at least one hardware project. Not because you'll necessarily work with FPGAs again, but because it breaks the illusion that software is the bottom of the stack. Understanding that everything you write eventually becomes electrons moving through transistors changes how you think about what your code is really doing.
