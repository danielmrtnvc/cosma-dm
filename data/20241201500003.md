"It works on my machine" to "it works for 10,000 users" is not a linear problem. There are decisions that feel fine at small scale that become catastrophic at larger scale, and vice versa -- things you'd never invest in for 100 users that are table stakes for 100,000.

The engineering decisions that haunt you later:

N+1 queries: your ORM makes one query to get a list of records, then a separate query for each record to get related data. With 10 records this is fine. With 10,000 this is a page that never loads. Most ORMs do this by default if you're not careful.

Missing indexes: covered in [[20241201100010|Databases]], but worth repeating. A query that works fine on a small table becomes brutally slow when the table grows. The time to add the index is before the table grows, not after.

No caching layer: hitting the database on every request works until it doesn't. A caching layer for frequently-accessed, infrequently-changing data is standard infrastructure for anything that needs to scale.

Synchronous processing for async work: if something doesn't need to happen in real-time, don't make the user wait for it. Queues and background workers let you do heavy work outside the request/response cycle.

The startup version of this: don't over-engineer for scale you don't have yet. Over-engineering is its own failure mode -- you spend time on infrastructure that you don't need instead of validating whether you have a product anyone wants. The art is knowing which technical decisions are reversible and which aren't, and only making the irreversible ones carefully.

If you use a shared database and it becomes a bottleneck, that's a good problem to have. It means you have enough users to create the problem. Optimizing for that before you have users is a trap.
